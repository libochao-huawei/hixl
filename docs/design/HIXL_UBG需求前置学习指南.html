1. 你要先理解什么

这个需求不是单纯新增一个字符串 ubg。它会穿过 HIXL 的三条主线：

  • 资源生成：从用户配置和设备信息生成本端 endpoint_list。
  • 建链选择：Client 从 Server 获取对端 endpoint 后，决定用 UBoE、UBG、UB、HCCS 还是 RoCE。
  • 单边传输：Host 内存需要先映射成 Device VA，再把 Device VA 注册给 Hcomm，并在传输前做地址翻译。
初学时建议不要从底层 Hcomm 开始钻。先看 EndpointConfig、EndpointGenerator、EndpointMatcher::MatchEndpoints 和 UBoE Host 内存注册路径，能更快理解 UBG 应该接在哪些点上。

2. 一页总览

可以先把这个需求想象成“在 PC16 液冷服务器上，基于 UBoE 已有框架新增 UBG 能力”。它不是重做一套链路，而是尽量复用 UBoE 的资源生成、建链和 Host 内存映射路径，并让自动选链知道什么时候应该用 UBG。

输入

用户配置

GlobalResourceConfig 写 ubg:device、LocalCommRes 显式给出 protocol=ubg，或者完全未配置时由 DSMI InterconType 自动探测 UBoE/UBG。

设备探测

类型判断 + EID 获取

先判断 ScaleOut 口 InterconType 是 UBoE 还是 UBG；是 UBG 后调用已有 DCMI/URMA EID 查询接口，把获取到的 EID 放入 local comm res。

输出

Endpoint

生成 protocol=ubg、comm_id=EID、placement=device 的 endpoint。

资源生成
EndpointGenerator 负责把 options 变成本端 endpoint_list。UBG 主要新增 ubg:device 识别、无配置时的 InterconType 自动探测，并复用现有 DCMI/URMA EID 查询能力生成默认 UBG endpoint。
协议选链
HixlClient 获取对端 endpoint 后，把匹配交给 EndpointMatcher::MatchEndpoints。UBG 主要新增 ScaleOut 匹配和三张优先级表。
CS 建链
HixlCSClient 向 Server 发 MatchEndpointReq，Server 侧 EndpointStore::MatchEndpoint 用底层 EndpointDesc 找 endpoint。
内存传输
Host 内存先 aclrtHostRegister 映射成 Device VA，再注册给 Hcomm；传输前 ConvertUboeDescs 类逻辑把 Host VA 转成 Device VA。UBG 与 UBoE 一致，建议抽象成“协议是否需要 Host 映射”的能力判断。
一句话记忆：UBG 开发要同时改“资源从哪里来”“链路怎么选”“Host 内存怎么给 Hcomm 用”这三件事。

3. 核心概念速查

概念 你可以怎么理解 和 UBG 的关系
EndpointConfig HIXL Engine 层使用的“通信资源描述”。字段包括 protocol、comm_id、placement、plane、dst_eid、net_instance_id。 新增 protocol="ubg"，comm_id 填 UBG EID。
EndpointDesc CS/Hcomm 层使用的底层 endpoint 描述，比 EndpointConfig 更接近 Hcomm 接口。 UBG 需要新增协议枚举，并把 EID 解析到 commAddr。
comm_id 不同协议的通信标识。RoCE/UBoE 常是 IP，UB/UBG 常是 EID，HCCS 是 device id。 UBG 的 comm_id 是调用已有 DCMI/URMA EID 查询接口获取到的 EID。
placement endpoint 在 host 还是 device 侧。 UBG 自动生成时固定为 device。
net_instance_id 超节点归属标识，用来判断本端和对端是否同超节点。 影响协议优先级：同超节点和跨超节点走不同选链表。
ScaleOut 口 超节点间扩展通信口，可配置为 UBoE 或 UBG。 UBG 需要先确认 ScaleOut 口类型为 UBG，再获取 EID。
InterconType 驱动接口返回的互联类型。 需要判断 ScaleOut 口是否配置为 UBG，查询有效代表物理链路可用。
DCMI / URMA 用于加载 DCMI、查询逻辑 ID、URMA 设备数量和 EID 列表的接口链路。 UBG 自动生成 endpoint 时通过它获取 EID 候选。
Mesh EID / CLOS PG EID Mesh EID 按 die/port 筛选；CLOS PG EID 是跨 Group 通信全局 EID 候选。 现有筛选服务 UB-CTP/CLOS local comm res；PC16 UBG 优先复用已有 EID 查询与解析链路，需确认返回 EID 可直接用于 UBG。

5. 通信资源与 endpoint

5.1 手动配置：LocalCommRes

LocalCommRes 是用户显式告诉 HIXL“本端有哪些通信资源”。1.3 格式核心是：

{
  "version": "1.3",
  "net_instance_id": "superpod1_1",
  "endpoint_list": [
    {
      "protocol": "roce",
      "comm_id": "192.168.1.10",
      "placement": "device"
    }
  ]
}

5.2 自动配置：GlobalResourceConfig

UBoE 当前已经支持通过 GlobalResourceConfig 自动生成 endpoint：

{
  "comm_resource_config.protocol_desc": ["uboe:device"]
}

UBG 需求要新增：

{
  "comm_resource_config.protocol_desc": ["ubg:device"]
}
如果业务什么都不配置，HIXL 也会按 DSMI InterconType 自动探测 ScaleOut 口类型：UBoE 取值生成 UBoE local comm res，UBG 取值生成 UBG local comm res。当前 RoCE 类取值不做无配置自动生成。

5.3 InterconType 取值

取值 含义 无配置自动生成策略
0 RoCE over DPU 当前不自动生成 RoCE
1 RoCE over CPU 当前不自动生成 RoCE
2 UBoE over NPU，David 直连出 UBoE 生成 UBoE local comm res
3 UBoE over SWITCH，Host 网卡出 RoCE 生成 UBoE local comm res
4 UBoE over DPU 生成 UBoE local comm res
待新增 UBG 生成 UBG local comm res

5.4 UBG endpoint 长什么样

{
  "version": "1.3",
  "net_instance_id": "superpod1_1",
  "endpoint_list": [
    {
      "protocol": "ubg",
      "comm_id": "000000000000004000100000dfdf1672",
      "placement": "device"
    }
  ]
}
注意：UBG 的 comm_id 是 EID，不是 IP。后续 ConvertToEndpointDesc 应把它按 EID 解析，EndpointDesc.commAddr.type 建议设置为 COMM_ADDR_TYPE_EID。

6. UBoE 现有业务逻辑

UBG 和 UBoE 很像，最值得先学的是 UBoE 现有路径。它已经解决了两个关键问题：自动资源生成和 Host 内存映射注册。

6.1 UBoE 自动资源生成

用户 options -> ParseConfigProtocolDesc() -> 检查 protocol_desc 是否包含 "uboe:device" -> BuildEndpointListFromLocalCommRes() -> 若没有显式 endpoint_list，则 GenDefaultUboeEndpointConfig() -> GetUboeIp() -> GetBondIpAddress() -> 生成 protocol=uboe, comm_id=bond IP, placement=device
函数/位置 用途
ParseConfigProtocolDesc
src/hixl/common/hixl_utils.cc		 解析 GlobalResourceConfig 中的 comm_resource_config.protocol_desc。
EndpointGenerator::BuildEndpointListFromOptions
src/hixl/engine/endpoint_generator.cc		 统一构建本端 endpoint 列表。
GenDefaultUboeEndpointConfig 生成默认 UBoE endpoint。
GetBondIpAddress
src/hixl/common/hixl_utils.cc		 调用 hccn_tool -g -ip -i <phy_id> -d bond<phy_id> 获取 Bond IP。

6.2 UBoE Host 内存注册

用户注册 Host 内存 -> Endpoint::RegisterMem() -> protocol == COMM_PROTOCOL_UBOE && mem.type == HOST -> HostRegisterProxy::RegisterByDev() -> 得到 registered_dev_mem -> HcommProxy::MemReg() 注册 Device VA -> HixlMemDesc.registered_dev_mem 记录映射后的 Device VA

这段逻辑在 src/hixl/cs/endpoint.cc。UBG 要复用这条路径，推荐把判断条件从“只认 UBoE”泛化为“需要 Host 映射的 ScaleOut 协议集合”。

7. DCMI/URMA 与 EID

UBG endpoint 的核心资源是 EID。PC16 UBG 基于 UBoE 增加能力，自动 local comm res 生成应复用已有 DCMI/URMA EID 查询链路：先确认 ScaleOut 口类型是 UBG，再调用 GetEidList 获取 EID，并写入 UBG comm_id。

7.1 EID 获取调用链

GenerateLocalCommRes -> BuildNpuRootinfos(related_npu_ids) -> for each npu_id: BuildNpuRootInfo(npu_id, is_server, root_info) -> GetUrmaDeviceList(npu_id) -> LoadUrmaDevicesFromDcmi(npu_id) -> DcmiProxy::LoadDcmi() -> DcmiProxy::GetLogicIdFromPhyId(npu_id, &logic_id) -> DcmiProxy::GetUrmaDeviceCnt(logic_id, &dev_cnt) -> for each urma_dev_index: DcmiProxy::GetEidList(logic_id, urma_dev_index, eid_list, &eid_cnt)

7.2 DCMI 接口职责

接口 功能 参数说明
LoadDcmi() 加载 libdcmi.so 通过 dlopen 动态加载。
GetLogicIdFromPhyId(phy_id, &logic_id) 物理 ID 转逻辑 ID 输入物理 NPU ID，输出逻辑 ID。
GetUrmaDeviceCnt(logic_id, &dev_cnt) 获取 URMA 设备数量 输入逻辑 ID，输出 URMA 设备数量。
GetEidList(logic_id, urma_dev_index, eid_list, &eid_cnt) 获取 EID 列表 输入逻辑 ID 和 URMA 设备索引，输出 EID 数组。

7.3 EID 数据结构与字符串

union DcmiUrmaEid {
  unsigned char raw[16];
  struct {
    unsigned long subnet_prefix;
    unsigned long interface_id;
  } in6;
};

struct DcmiUrmaEidInfo {
  DcmiUrmaEid eid;
  unsigned int eid_index;
};

DCMI 返回 16 字节原始 EID。HIXL 侧应按原始字节序转换为 32 位十六进制字符串，不带冒号分隔符。

原始显示: 0000:0000:007f:0200:0010:0000:df00:9001 字符串: 00000000007f020000100000df009001

7.4 现有 EID 筛选逻辑

byte6

从字符串索引 10-11 取第 6 字节

例如 f2 转为 0xf2，高半字节用于判断 die/PG，低半字节表示 port。

die_id

高半字节 bit2

die_id = (high_nibble & 0x4) ? 1 : 0。

PG EID

高半字节 0x3 或 0x7

满足 high_nibble == 0x3 || high_nibble == 0x7 时为 PG EID。

port

低半字节

port = low_nibble。

EID 类型 筛选条件 用途
Mesh EID 非 PG EID，且 die_id == mesh_die_id。Pod 0-3 组用 die0，4-7 组用 die1；Server 固定 die1。 写入 port_to_eid["die/port"]。
CLOS PG EID PG EID。 作为跨 Group 通信的全局 EID 候选，写入 clos_pg_eids。 
struct NpuRootInfo {
  std::map<std::string, std::string> port_to_eid; // "die/port" -> Mesh EID
  std::vector<ClosPgEid> clos_pg_eids;            // CLOS PG EID 列表
};
注意：现有 CollectMeshPorts / CollectClosPgEids 服务 UB-CTP/CLOS local comm res。UBG 优先复用已有 DCMI EID 查询、字符串转换和 byte6 解析能力；若直接复用筛选结果，需要和底层确认该 EID 可直接作为 UBG comm_id。

8. 建链流程

8.1 Engine 层：选协议

HixlEngine::Connect() -> ClientManager::CreateClient() -> HixlClient::Initialize() -> Client socket 请求 Server endpoint_list -> EndpointMatcher::MatchEndpoints() -> ClientHandlerFactory::Create() -> 创建 DIRECT 或 UB handler

当前 EndpointMatcher::MatchEndpoints 的实际逻辑：

  1. 先尝试 UBoE：TryMatchUboe()。
  2. 如果 MustUseRoce() 命中，强制匹配 RoCE 并返回。
  3. 否则尝试 UB：TryMatchUb()。
  4. UB 无匹配时尝试 HCCS：TryMatchHccs()。
容易误解：RoCE 不是普通同超节点路径的第二优先级，而是 HCCL_INTRA_ROCE_ENABLE=1 或 net_instance_id 不同触发的强制分支。UBG 需求会重新定义跨超节点和同超节点的遍历表。

8.2 CS 层：匹配 EndpointDesc 并创建 channel

HixlCSClient::Connect() -> SendMatchEndpointRequest(remote_endpoint_) -> Server: EndpointStore::MatchEndpoint() -> 返回 remote endpoint handle -> SendCreateChannelRequest() -> Endpoint::CreateChannel()

Engine 层选择的是“用哪种协议建链”，CS 层匹配的是“这个底层 endpoint 是否存在”。UBG 开发时两层都要考虑。

8.3 一次 Connect 的完整时序

  1. Server 初始化 endpoint

    HixlServer 把本端 endpoint_list 转成 EndpointDesc，交给 CS 层创建 endpoint，并保存在 EndpointStore。

  2. Client 获取对端资源

    HixlClient::Initialize 通过控制面 socket 发 GetEndpointInfoReq，Server 返回序列化后的 endpoint_list。

  3. Engine 层匹配协议

    EndpointMatcher::MatchEndpoints 根据本端和对端 EndpointConfig 决定匹配结果，例如 UBG 对 UBG、UBoE 对 UBoE、RoCE 对 RoCE。

  4. 创建 handler

    ClientHandlerFactory 根据匹配结果创建 DIRECT 或 UB handler。UBG 属于 ScaleOut direct 类协议，预期走 direct handler。

  5. CS 层匹配底层 endpoint

    HixlCSClient 发送 MatchEndpointReq，Server 侧 EndpointStore::MatchEndpoint 用 EndpointDesc 查找实际 endpoint。

  6. 创建 channel

    匹配成功后再发 CreateChannelReq，底层 Endpoint::CreateChannel 创建 Hcomm channel，之后才能注册远端内存并传输。

9. Host 内存与 VA 转换

UBG 单边通信需求中最核心的传输要求是：Host 内存要映射到 Device 侧，再把 Device VA 注册给 Hcomm，传输前还要把用户传入的 Host VA 翻译成 Device VA。

9.1 数据结构

结构/字段 用途
HixlMemDesc::registered_dev_mem 记录 Host 内存映射后的 Device VA，当前 UBoE 场景有效，UBG 可复用。
MemoryRegion::register_dev_addr HixlMemStore 中记录某段注册内存对应的 Device VA。
HixlOneSideOpDesc CS Client 传输时使用的单边操作描述，包含 remote_buf、local_buf 和长度。

9.2 UBoE 当前转换路径

BatchTransfer() / BatchTransferSync() -> ValidateAddress() 仍用用户 Host VA 校验 -> Device endpoint 且 protocol == UBOE -> ConvertUboeDescs() -> ConvertHostRegisterAddr() -> device_addr = register_dev_addr + (host_addr - region.addr) -> BatchTransferDevice() / BatchTransferDeviceSync() -> 下发 AICPU kernel

UBG 应把这里的协议判断从 COMM_PROTOCOL_UBOE 扩展为统一能力判断，例如 IsHostRegisterMappedProtocol。初始集合包含 UBoE 和 UBG，后续有新协议需要 Host 映射时直接加入集合；函数可以考虑泛化为 ConvertScaleOutDescs。

9.3 Host 内存为什么要映射成 Device VA

用户视角

API 里仍然传 Host VA

上层调用注册和传输时，看到的是原始 Host 地址。这保证业务侧不用感知 Device VA。

Hcomm 视角

底层要使用 Device VA

UBoE/UBG 这类 Device endpoint 传输时，Hcomm/AICPU kernel 需要能在 Device 侧访问这段内存。

注册阶段

记录映射关系

Endpoint::RegisterMem 通过 HostRegisterProxy::RegisterByDev 得到 registered_dev_mem，并保存在内存描述中。

传输阶段

按 offset 转换地址

ConvertHostRegisterAddr 查到注册区域后，用 registered_dev_mem + offset 替换传输描述中的地址。

举例： Host 注册基址 region.addr = 0x10000000 Host 传输地址 remote_buf = 0x10001000 映射后 Device 基址 register_dev_addr = 0x80000000 offset = 0x10001000 - 0x10000000 = 0x1000 最终下发给 Hcomm/AICPU 的地址 = 0x80000000 + 0x1000 = 0x80001000

10. 协议选择逻辑

需求指定的新优先级应按场景理解：

场景 优先级 解释
指定 RoCE 环境 RoCE HCCL_INTRA_ROCE_ENABLE=1 或业务明确指定 RoCE 时，直接只走 RoCE。
超节点间 UBoE / UBG / RoCE 未命中强制 RoCE 时，跨超节点优先尝试 ScaleOut 协议，最后 RoCE。
超节点内 UB / HCCS / UBoE / UBG / RoCE 同超节点优先走超节点内 UB，失败后 HCCS，再尝试 ScaleOut，最后 RoCE。
已确认策略：指定 RoCE 优先级最高。命中后即使业务显式配置 UBG/UBoE，也只尝试 RoCE。

10.1 新优先级决策表

第一问
是否指定 RoCE 环境？例如 HCCL_INTRA_ROCE_ENABLE=1。
是：只匹配 RoCE
第二问
是否显式指定 UBG/UBoE？注意第一问为“否”时才看这里。
是：只匹配指定协议
第三问
本端与对端 net_instance_id 是否不同？
是：UBoE / UBG / RoCE
第四问
同超节点普通场景。
走：UB / HCCS / UBoE / UBG / RoCE

UB 匹配规则

当前 UB 匹配通过 BuildMatchMap() 和 TryMatchUb() 实现。对端 map key 是 {dst_eid, plane, placement}，本端查询 key 是 {comm_id, plane, placement}。这等价于优先满足“对端 dst_eid 匹配本端 comm_id”。

如果拆分为 TryMatchUbByDstEid 和 TryMatchUbByPlane，要保持 MatchKey::Matches 中 dst_eid 为空时按 plane/placement 降级的通配语义。

11. UBG 需求改动点

11.1 配置和资源自动获取

  • 新增协议字符串 kProtocolUbg = "ubg"。
  • GlobalResourceConfig 支持 "ubg:device"。
  • EndpointGenerator 新增 UBG 默认 endpoint 生成逻辑。
  • 新增 ScaleOut 类型判断 wrapper：查询 InterconType，确认 ScaleOut 口为 UBG。
  • 复用现有 DCMI/URMA EID 获取、字符串转换和 byte6 解析逻辑；将获取到的 EID 写入 UBG comm_id。
  • 配置 ubg:device 但 InterconType 返回非 UBG 时，返回可诊断错误。

11.2 EndpointDesc 转换

  • 新增 COMM_PROTOCOL_UBG，具体枚举值需要和 Hcomm/驱动确认。
  • protocol=ubg 映射到 COMM_PROTOCOL_UBG。
  • comm_id 按 EID 解析，commAddr.type 应设置为 COMM_ADDR_TYPE_EID。
  • placement=device 时填充物理 device 信息。

11.3 建链匹配

  • Engine 层新增 UBG 匹配函数，例如 TryMatchUbg() 或泛化 TryMatchScaleOut()。
  • 按“指定 RoCE / 超节点间 / 超节点内”三张表重构 EndpointMatcher::MatchEndpoints()。
  • ScaleOut 匹配要求本端与对端 protocol 一致：UBG 对 UBG，UBoE 对 UBoE。
  • 当前设计不要求两端 EID 相等；EID 传给 Hcomm，是否按 EID 关系匹配待驱动语义确认。

11.4 Host 内存与传输

  • Endpoint::RegisterMem() 中，把 UBoE Host register 条件泛化到 UBG。
  • HixlCSClient::BatchTransferAsync() 和 BatchTransferSync() 中，把 UBoE 地址转换条件泛化到 UBG，并将 ConvertUboeDescs 泛化为 ConvertScaleOutDescs。
  • HixlMemStore 已具备记录 Host 内存和 Device 映射地址的字段，优先复用。

11.5 CS 层 EndpointStore

EndpointStore::MatchEndpoint() 当前对 HCCS、UB、RoCE 做专门地址比较，其它协议在 protocol 一致后直接匹配。UBG 初期可沿用 protocol-only 行为，避免错误要求两端本端 EID 相等；若驱动后续明确 EID 对端关系，再增加 UBG 专属比较。

12. 端到端开发视图

下面按一次“自动获取 UBG 资源并完成 Host 内存传输”的路径，把你需要改的点串起来。

  1. 配置进入 HIXL

    业务设置 GlobalResourceConfig={"comm_resource_config.protocol_desc":["ubg:device"]}。你需要让解析逻辑识别 ubg:device，并避免和 uboe:device 同时出现。

  2. 生成本端 UBG endpoint

    EndpointGenerator 判断没有显式 endpoint_list 后，先查询 InterconType，再通过 DCMI/URMA 获取并校验 EID，生成 protocol=ubg 的 EndpointConfig。

  3. Server 初始化底层 endpoint

    ConvertToEndpointDesc 把 protocol=ubg 转为 COMM_PROTOCOL_UBG，把 EID 写入 commAddr，再交给 CS Server 创建 endpoint。

  4. Client 获取对端 endpoint 并选链

    HixlClient 拉取 Server endpoint 后，EndpointMatcher::MatchEndpoints 按新优先级表选择 UBG、UBoE、UB、HCCS 或 RoCE。

  5. CS 层确认 endpoint 存在

    HixlCSClient 在创建 channel 前发送 MatchEndpointReq。Server 侧 EndpointStore::MatchEndpoint 确认对应 UBG endpoint 存在。

  6. Host 内存注册

    用户注册 Host 内存时，UBG 和 UBoE 一样先 aclrtHostRegister 得到 Device VA，再把 Device VA 注册给 Hcomm，并保存 registered_dev_mem。

  7. 传输前地址转换

    传输时 ConvertScaleOutDescs 根据 HixlMemStore 中记录的区域，把 remote_buf 和 local_buf 中的 Host VA 转成 Device VA。

  8. AICPU kernel 执行传输

    转换后的地址随 HixlOneSideOpDesc 下发给 Device 路径，最终由 AICPU kernel 调用 Hcomm 完成单边读写。
