## Environment Requirements

### 1. Hardware and Software Preparation

- Chips: Atlas A3 training/inference products, Atlas 800I A2 inference products/A200I A2 Box heterogeneous components, Ascend 950PR/Ascend 950DT
- Refer to [Environment Setup](../docs/en/build.md#environment-setup) to complete Ascend AI software stack deployment in the runtime environment

### 2. Device Connectivity Check

Before running samples, use the [hccn_tool](https://support.huawei.com/enterprise/en/ascend-computing/ascend-hdk-pid-252764743?category=developer-documents&subcategory=interface-reference) provided in the driver package to check **connectivity between two devices**. Using A2 scenario as an example:

> If the `hccn_tool` command cannot be found, search for the executable file in the CANN driver package installation directory (default `/usr/local/Ascend/driver/tools/hccn_tool`), and use `ln -s` to add it to `PATH`.

- Step 1: Query the IP information of required devices, using an 8-card setup as an example:

```shell
for i in {0..7}; do hccn_tool -i $i -ip -g; done
```

- Step 2: Check connectivity between two devices, using devices a and b as an example:

```shell
hccn_tool -i ${device_id_a} -ping -g address ${ip_address_b}
hccn_tool -i ${device_id_b} -ping -g address ${ip_address_a}
```

If `recv time out seq` is returned, the two devices are not connected.

- RDMA between A3 single-card dual-die may not work; even after environment configuration, `ping` may fail. The most accurate way to determine is using `roce_test ib_send_bw` for traffic testing:

```bash
# Receiver
/usr/local/Ascend/driver/tools/hccn_tool -i 0 -roce_test reset
/usr/local/Ascend/driver/tools/hccn_tool -i 0 -roce_test ib_send_bw -s 65536 -n 1000 -tcp

# Sender
/usr/local/Ascend/driver/tools/hccn_tool -i 1 -roce_test reset
PEER_IP=$(/usr/local/Ascend/driver/tools/hccn_tool -i 0 -ip -g 2>/dev/null | sed -n 's/^ipaddr:\(.*\)/\1/p' | head -1)
/usr/local/Ascend/driver/tools/hccn_tool -i 1 -roce_test ib_send_bw -s 65536 -n 1000 address "$PEER_IP" -tcp
```

- Step 3: Check TLS certificate configuration consistency between devices:

```shell
for i in {0..7}; do hccn_tool -i $i -tls -g; done | grep switch
```

Devices with TLS enabled cannot establish connections with devices where TLS is disabled. Example (disable TLS):

```shell
for i in {0..7}; do hccn_tool -i $i -tls -s enable 0; done
```

If `hccn_tool is busy, please try again` appears, avoid concurrent use of the command and retry later.

### Constraint Notes

- **HCCS**: On **A2 (Ascend910B-class)** only **D2D** (`--type=D2rD` / `rD2D`); on **A3 (Ascend910-class)** also supports **H2rD** / **rD2H**.

## Quick Start

### 1. Build

Run in the repository root directory:

```bash
bash build.sh --examples
```

Compiled executables are in `build/benchmarks/`:

- `comm_benchmark/hixl_comm_bench` — Communication performance test
- `kv_benchmark/hixl_kv_bench` — KV Cache scenario test

### 2. Run All Tests with One Command

```bash
bash benchmarks/run_all_bench.sh
```

This script automatically:

- Checks and sources CANN environment
- Detects chip model through `npu-smi info` (A2 / A3 / A5; device name containing Ascend910B→A2, Ascend910→A3, Ascend950→A5)
- Runs all supported communication direction × transmission type combinations
- Runs KV Cache benchmarks
- Generates `benchmarks/perf.md` and line charts
- Prints performance results in terminal

After completion, open `benchmarks/perf.md` to view results.

### 3. Custom Parameters

```bash
# Custom communication benchmark repetition loops and devices
bash benchmarks/run_all_bench.sh --loops 10 --device-ids 0,1,2,3,4,5,6,7

# Run only KV tests
bash benchmarks/run_all_bench.sh --skip-comm

# Run only communication tests
bash benchmarks/run_all_bench.sh --skip-kv

# Pass HIXL Initialize() options to communication benchmark (consistent with hixl_comm_bench's -H=KEY=VALUE, can be repeated)
bash benchmarks/run_all_bench.sh --hixl-option 'LocalCommRes={"version":"1.3"}'
```

### 4. Performance Data Summary

- **`perf.md`**: Automatically generated by `run_all_bench.sh` on a single platform (A2 / A3 / A5), containing current platform performance tables and line charts.
- **`performance.md`**: Multi-platform summary document (organized by A2 / A3 / A5 chapters), manually maintained by developers, content sourced from `perf.md` generated on each platform.

## Communication Benchmark (`hixl_comm_bench`)

Measures HIXL block transmission bandwidth under different directions and transmission types. Bandwidth data in "GB/s" uses **decimal** definition: **1 GB = 10⁹ bytes**.

**Concepts**:

- **Initiator**: The party that initiates transmission (read / write)
- **Target**: The party that responds to transmission (registers memory, waits for initiator connection)
- **Direction**: Determined by Initiator memory type + Target memory type + operation type

### Direction Naming

Direction name format is `source → remote target`, where **D**=Device, **H**=Host, **r**=remote.

| Direction | Meaning | Operation |
| :--- | :--- | :---: |
| **D2rD** | Device writes to remote Device | write |
| **rD2D** | Read from remote Device to Device | read |
| **D2rH** | Device writes to remote Host | write |
| **rH2D** | Read from remote Host to Device | read |
| **H2rH** | Host writes to remote Host | write |
| **rH2H** | Read from remote Host to Host | read |
| **H2rD** | Host writes to remote Device | write |
| **rD2H** | Read from remote Device to Host | read |

### Single-Machine Run

```bash
# Quick test for one direction
python3 benchmarks/comm_benchmark/scripts/run_comm_benchmark.py --type=D2rD --transport=hccs

# Specify devices and ports
python3 benchmarks/comm_benchmark/scripts/run_comm_benchmark.py \
  --type=D2rH --transport=rdma --device_ids=0,1

# One-to-many pattern
python3 benchmarks/comm_benchmark/scripts/run_comm_benchmark.py \
  --pattern=one_to_many --device_ids=0,1,2,3,4 \
  --type=D2rD --transport=hccs

# Many-to-one pattern
python3 benchmarks/comm_benchmark/scripts/run_comm_benchmark.py \
  --pattern=many_to_one --device_ids=0,1,2,3,4 \
  --type=D2rD --transport=hccs

# Pass HIXL Initialize options (same as hixl_comm_bench's -H=KEY=VALUE, can use -H multiple times)
python3 benchmarks/comm_benchmark/scripts/run_comm_benchmark.py \
  --type=D2rD --transport=hccs \
  -H 'LocalCommRes={"version":"1.3"}'
```

### Dual-Machine Run

**1:1 (default)**:

```bash
python3 benchmarks/comm_benchmark/scripts/run_comm_benchmark.py \
  --role=target --transport=rdma
# initiator: copy the command printed by target
```

**one_to_many** (target multiple NPUs, initiator single NPU):

```bash
python3 benchmarks/comm_benchmark/scripts/run_comm_benchmark.py \
  --role=target --transport=rdma --pattern=one_to_many --device_ids=0,1,2
# initiator: copy the command printed by target
```

**many_to_one** (target single NPU, initiator multiple NPUs):

```bash
python3 benchmarks/comm_benchmark/scripts/run_comm_benchmark.py \
  --role=target --transport=rdma --pattern=many_to_one --num_initiators=3
# initiator: copy the command printed by target
```

### Key Parameters

| Parameter | Description | Options | Default |
|---|---|---|---|
| `--type` | Transmission direction | `D2rD`, `rD2D`, `D2rH`, `rH2D`, `H2rH`, `rH2H`, `H2rD`, `rD2H` | `D2rD` |
| `--transport` | Transmission path | `hccs` / `rdma` / `fabric_mem` / `uboe` / `ubg` / `ub` | `hccs` |
| `--pattern` | Communication topology | `pairwise` / `one_to_many` / `many_to_one` | `pairwise` |
| `--start_block_size` | Starting block size (bytes) | Integer | 16384 (16K) |
| `--max_block_size` | Maximum block size, doubles incrementally | Integer | 2097152 (2M) |
| `--loops` | Number of repeated runs | Positive integer | 5 |
| `--device_ids` | Device ID list to use | Comma-separated | `0,1` |
| `--plot` | Generate PNG charts for new or updated CSV in this run | Switch | Enabled |
| `--skip_plot` | Skip PNG chart generation | Switch | Disabled |
| `--soc_variant` | SOC passed to `hixl_comm_bench`: `auto` / `a2` / `a3` / `a5` | `auto`, `a2`, `a3`, `a5` | Auto-detected by default |
| `-H` / `--hixl_option` | Initialization options passed to `hixl_comm_bench` | `KEY=VALUE`, repeatable | (None) |

### Support Matrix

| Platform | HCCS | ROCE (RDMA) | FabricMem |
|---|---|---|---|
| **A2** | D2rD, rD2D | All 8 directions | Not supported |
| **A3** | D2rD, rD2D, H2rD, rD2H | All 8 directions | All 8 directions |

## KV Benchmark (`hixl_kv_bench`)

Simulates KV pooling scenario, tests put/get performance based on model shapes and KV block counts.

### Model Support

| Model | Layers | Attention Type | KV Strategy | Description |
|---|---|---|---|---|
| `deepseek-r1` | 61 | MLA | shared | Equal MLA cache per key |
| `glm5` | 78 | MLA + DSA | shared | Equal MLA + DSA cache per key |
| `deepseek-v4` | 61 | Hybrid CSA/HCA + SWA | shared | SWA (`max_key_count=1`) only transfers key0, one copy per layer |

**shared strategy**: MLA models share the same KV Cache across all inference ranks. In testing, rank 0 is responsible for writing (put) all keys, and all ranks read (get) in parallel.

The `total_bytes` / `total_transfer_bytes` in logs and CSV/JSON represents **the actual transmission byte sum of all keys under this workload** (summarized by slice, respecting `max_key_count`), not "single key size × key_count".

### Running Examples

```bash
python3 benchmarks/kv_benchmark/scripts/run_kv_benchmark.py \
  --model=deepseek-r1 \
  --transport=fabric_mem

# More detailed parameters
python3 benchmarks/kv_benchmark/scripts/run_kv_benchmark.py \
  --num_processes=8 \
  --devices=0,1,2,3,4,5,6,7 \
  --model=deepseek-r1 \
  --key_counts=16,32,48,64 \
  --transport=fabric_mem
```

### KV Parameters

| Parameter | Description | Default |
|---|---|---|
| `--model` | Model configuration name | `deepseek-r1` |
| `--key_counts` | KV block/key counts to test, comma-separated | `16,32,48,64` |
| `--num_processes` | Number of concurrent processes (simulating inference rank count) | `8` |
| `--transport` | Transmission path | `fabric_mem` |
